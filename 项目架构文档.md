## 1. 项目概述

**SenseJourney (逃生舱)** 是一款专为高压都市人群设计的**"盲盒式周末免决策逃跑服务"**。本产品彻底摒弃传统旅游 App 的"工具属性"与"比价逻辑"，转而提供极致省心、端到端全包的避世体验。

### 核心价值主张
- **去选择化**：用户仅输入客观限制条件，系统输出唯一的打包方案
- **全包式闭环交付**：方案包含精选非标住宿、餐饮安排及特定情绪场景
- **反焦虑的盲盒机制**：购买环节隐藏具体目的地，通过渐进式时间轴机制解锁线索

## 2. 技术栈

### 前端技术栈
- **框架**: React Native + Expo
- **状态管理**: React Hooks (useState, useEffect, useContext)
- **导航**: React Navigation (Stack Navigator)
- **动画**: React Native Animated API
- **样式**: React Native StyleSheet
- **平台**: 支持 Web、iOS、Android 三端
- **HTTP客户端**: 原生 fetch API

### 后端技术栈
- **语言**: Go 1.21+
- **Web框架**: Gin
- **ORM**: GORM
- **数据库**: MySQL (主库) + SQLite (开发备用)
- **ID系统**: Snowflake-like 算法 + Hashids 编码
- **依赖管理**: Go Modules

### 数据库
- **主数据库**: MySQL
  - 连接字符串: `root:password@tcp(localhost:3306)/sensejourney?charset=utf8mb4&parseTime=True&loc=Local`
  - 环境变量覆盖: `MYSQL_DSN`
- **开发备用**: SQLite (当MySQL不可用时自动降级)

### 3.2 核心架构特性

1. **前后端分离架构**
   - 前端: React Native 跨平台应用
   - 后端: Go RESTful API 服务
   - 通信: JSON over HTTP/HTTPS

2. **ID系统设计**
   - **后端存储**: uint64 数字ID (Snowflake-like算法生成)
   - **前端传输**: Hashids 编码的短字符串
   - **算法组成**: 41位时间戳(5分钟块) + 12位机器ID + 11位序列号

3. **数据库设计**
   - **主外键关系**: users.uid ↔ journey_selections.uid
   - **索引优化**: uid字段建立索引，支持快速查询
   - **迁移策略**: 支持从UUID到数字UID的无缝迁移

4. **错误处理与降级**
   - MySQL连接失败时自动降级到SQLite
   - GitHub Pages环境提供模拟数据支持
   - 详细的错误日志和用户友好的错误提示

## 5. 数据库设计

### 5.1 users 表
```sql
CREATE TABLE users (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    uid BIGINT UNSIGNED UNIQUE NOT NULL,          -- 数字UID (Snowflake算法生成)
    phone VARCHAR(20),
    email VARCHAR(255),
    password VARCHAR(255),                        -- 加密存储
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_uid (uid)                           -- 查询优化索引
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 5.2 journey_selections 表
```sql
CREATE TABLE journey_selections (
    id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    uid BIGINT UNSIGNED NOT NULL,                 -- 关联用户UID
    budget VARCHAR(50) NOT NULL,                  -- 预算范围
    duration VARCHAR(50) NOT NULL,                -- 持续时间
    proximity VARCHAR(50) NOT NULL,               -- 距离范围
    transport VARCHAR(50) NOT NULL,               -- 交通方式
    moods JSON NOT NULL,                          -- 情绪标签数组
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_uid (uid),                          -- 用户查询优化
    INDEX idx_created_at (created_at),            -- 时间排序优化
    FOREIGN KEY (uid) REFERENCES users(uid) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

### 5.3 数据关系
- **1:N关系**: 一个用户可创建多个旅程选择
- **外键约束**: journey_selections.uid → users.uid
- **级联删除**: 用户删除时自动删除其所有旅程选择

## 6. API接口规范

### 6.1 通用约定
- **请求格式**: application/json
- **响应格式**: JSON
- **身份验证**: 通过uid参数传递用户标识
- **错误码**: HTTP状态码 + 错误消息

## 7. 部署与运行

### 7.1 本地开发环境

#### 后端启动
```bash
cd backend
go run main.go
# 服务运行在 http://localhost:8080
```

#### 前端启动
```bash
npx expo start --web --port 8082
# 浏览器访问 http://localhost:8082
```

### 7.2 数据库配置

#### MySQL配置（推荐）
```bash
# 创建数据库
mysql -u root -p
CREATE DATABASE sensejourney CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

# 使用环境变量（可选）
export MYSQL_DSN="root:password@tcp(localhost:3306)/sensejourney?charset=utf8mb4&parseTime=True&loc=Local"
```

#### SQLite配置（开发备用）
- 无需额外配置，当MySQL不可用时自动创建SQLite文件

### 7.3 生产环境部署

#### 后端部署
1. 编译Go应用: `cd backend && go build -o sensejourney-api main.go`
2. 配置环境变量: `MYSQL_DSN`, `PORT`等
3. 使用systemd或docker运行服务

#### 前端部署
1. Web构建: `npx expo build:web`
2. 部署输出目录: `web-build/`
3. 配置API代理或CORS

#### GitHub Pages部署
- 前端已适配GitHub Pages环境
- 自动检测github.io域名
- 提供模拟数据支持离线演示

## 9. 故障排除

### 9.1 常见问题

#### MySQL连接失败
```
错误: Error 1045: Access denied for user 'root'@'localhost' (using password: YES)
解决: 检查密码是否正确，或使用环境变量MYSQL_DSN覆盖默认配置

```
错误: listen tcp :8080: bind: address already in use
解决: 停止占用端口的进程或修改端口配置
```

#### 前端API调用失败
```
错误: Network request failed
解决: 检查后端服务是否运行，CORS配置是否正确

## 附录

### A. 技术决策说明

1. **为什么选择Go作为后端语言?**
   - 高性能，编译型语言
   - 优秀的并发支持
   - 简洁的语法和强大的标准库
   - 适合构建高并发的API服务

2. **为什么使用Snowflake-like算法而不是UUID?**
   - 数字ID更适合数据库索引性能
   - 时间有序，利于分片和排序
   - 通过Hashids编码后仍然保持URL友好
   - 避免UUID的随机性带来的索引碎片

3. **为什么支持MySQL和SQLite双数据库?**
   - MySQL适合生产环境，性能好，功能完整
   - SQLite适合开发和测试，零配置
   - 自动降级机制提高系统容错性
